

The [tutorial YAML](https://github.com/mir-group/allegro/blob/main/configs/tutorial.yaml) is a ready-made training recipe for a small Allegro model inside the NequIP framework. Conceptually, it wires up four things: a dataset of atomic structures and reference labels (energies/forces), the Allegro architecture and its local neighbor cutoff, the optimization loop (loss terms and optimizer), and the run logistics (device, batch size, logging, checkpoints). The Allegro repo itself points to this file as the canonical “smallest useful” config for getting a model to train, and the official docs say you train “within the NequIP framework,” with Allegro supplied as the model block in that YAML. ([GitHub](https://github.com/mir-group/allegro "GitHub - mir-group/allegro: Allegro is an open-source code for building highly scalable and accurate equivariant deep learning interatomic potentials"))

What the example is trying to accomplish is a quick, end-to-end fit: read a modest dataset; build graphs where each atom connects only to neighbors within a cutoff radius; push those graphs through a strictly-local, E(3)-equivariant network (Allegro) to predict per-atom energies and forces; and minimize a loss that mixes force error (dominant) and total energy error (small but stabilizing). Allegro’s “strictly local” design means it doesn’t pass messages across the whole graph the way classic NequIP does; instead, it stacks equivariant tensor products locally, which is why it scales so well to large systems—but for a toy dataset the effect you’ll notice is just that the model trains smoothly on CPU too. If you like to peek under the hood, the high-level properties and motivations are summarized in the Allegro overview and paper; the YAML’s model section simply picks hyperparameters for that architecture. ([nequip.readthedocs.io](https://nequip.readthedocs.io/projects/allegro?utm_source=chatgpt.com "Allegro - NequIP documentation - Read the Docs"))

How it works at run-time is straightforward. `nequip-train` parses the YAML, builds a data loader that constructs neighbor lists with the specified cutoff, instantiates the Allegro model with the listed chemical species and representation widths, and then runs minibatches through PyTorch. Each step computes predicted forces by differentiating the predicted energy with respect to positions; the loss is a weighted sum over force and energy errors. The optimizer (typically Adam in the tutorial) updates weights; a simple scheduler may reduce the learning rate as training plateaus. NequIP handles bookkeeping: splitting the dataset, shuffling, gradient accumulation if requested, and checkpointing. The “device” key decides whether tensors live on CPU or CUDA; nothing else about the pipeline changes when you choose CPU. The maintainers’ guidance to use NequIP’s training entry point with Allegro configured in YAML is exactly what you’ll be exercising here. ([nequip.readthedocs.io](https://nequip.readthedocs.io/projects/allegro/en/latest/guide/guide.html?utm_source=chatgpt.com "User Guide - Allegro documentation"))

When you run the tutorial, the concrete outputs land in a run directory named after the `run_name` (and/or a timestamp). You should expect a `model.pth` (the trained weights), a `config.yaml` copy frozen with all defaults resolved, a `metadata.yaml` with environment and git info, training logs (plain text and JSON), periodic checkpoints (e.g., `epoch-*.pth`), and learning-curve files (CSV/JSON). If the config includes a test split, you’ll also get a final metrics dump with MAEs/RMSEs for forces and energies. These are the standard NequIP artifacts; Allegro piggybacks on this machinery. The Allegro README explicitly frames Allegro as a NequIP extension and sends you to the NequIP docs for “how to train, test, and use models,” which is why the output shape matches NequIP’s. ([GitHub](https://github.com/mir-group/allegro "GitHub - mir-group/allegro: Allegro is an open-source code for building highly scalable and accurate equivariant deep learning interatomic potentials"))

There is one version footnote worth calling out before you hit enter. The Allegro repository notes a major, backwards-incompatible NequIP update in April 2025 (NequIP v0.7.x + Allegro v0.4.x). Your environment from the smoke test shows NequIP 0.6.1 and Allegro 0.3.0, i.e., the pre-0.7 API. If you are reading the tutorial YAML from the current `main` branch, some field names or defaults may target v0.7+ semantics. If that happens, either run the tutorial from the corresponding Allegro tag that matches your versions or adapt a couple of keys to the 0.6-series schema; the README explicitly flags this compatibility boundary. ([GitHub](https://github.com/mir-group/allegro "GitHub - mir-group/allegro: Allegro is an open-source code for building highly scalable and accurate equivariant deep learning interatomic potentials"))

Running on CPU is entirely fine for this example. Point the device to CPU—either by adding `device: cpu` in the YAML or overriding on the command line—and launch:

```bash
nequip-train configs/tutorial.yaml device=cpu wandb.mode=disabled
```

On a modest CPU node the epochs will simply take longer. You’ll still see loss values decreasing, periodic checkpoint saves, and a final model. If you later repeat the exact same run on a GPU, the numerical results will be the same up to usual stochastic differences; the only practical change is wall-clock time.

In short, the minimal example is a compact, batteries-included training script in YAML form. Its goal is to prove your Allegro+NequIP toolchain from dataset → model → trained weights, with a simple local-equivariant network and a standard force-dominated loss. It operates by constructing neighbor-cutoff graphs, running them through the Allegro blocks, and optimizing with Adam, and it emits the usual NequIP artifacts—checkpoints, logs, metrics, and a `model.pth` you can deploy. If you keep your `allegro`/`nequip` versions aligned with the example’s branch, you can run it today on CPU and expect all of the above to appear in your run directory.
